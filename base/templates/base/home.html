Stock report web app

{% blocktrans %}The goal was to first collect trading data by scraping CSV files from a specific website and display it on a website in a dashboard.{% endblocktrans %}

{% blocktrans %}The data is collected every work day and saved in a postgres database. Currently there is about 5 million trades processed daily. After scraping is done the aggregated statistics are calculated for every Stock (calculated fairly quickly with raw SQL).{% endblocktrans %}

{% blocktrans %}On the front-end, the tables are based on Datatables connected to the API from django. Feature wise there is also filtering/exporting feature to XLSX.{% endblocktrans %}

Processing German Lottery

{% blocktrans %}Client's goal was to periodically get data from three different kinds of lotteries in Germany and load it into a MySQL database for further processing.{% endblocktrans %}

{% blocktrans %}Client prepared the initial database design, which needed to be tweaked a bit by me (changing few column types, adding unique index and so on). The main table has 50+ columns with mostly numerical data, which differ and overlap in use within the 3 different lotteries. Large challenge was processing the structured file properly and fixing any issues which arised due to inconsistencies within the data.{% endblocktrans %}

{% blocktrans %}Large focus of the project was configurability and flexibility of the script as it will run on schedule on the server. Script settings were separated into its own file for simpler usage. I also focused on making logging work seamlessly for both output to the console&syslog.{% endblocktrans %}

{% blocktrans %}I really liked this project. The data was new to me and it was a interesting kind of a challenge.{% endblocktrans %}

Processing historic Bitcoin data from Poloniex

{% blocktrans %}Client first wanted me to import historic financial data from Poloniex (cryptocurrency market) API into a SQLite database. I had to ensure that the script pulls up complete history of trading for a specific symbol (currency pair like Bitcoin-Ethereum). The API responds with batches of trades of 50k and my script goes back as far as possible to get all the data without any duplicates into the database.{% endblocktrans %}

{% blocktrans %}After some discussion we agreed to switch to MySQL due to the amount of data and possible complications later down the line if we were stuck with SQLite. At the time I've processed about 4.5 million trades for Bitcoin-Ethereum currency pair which took ~700MB in the database.{% endblocktrans %}

{% blocktrans %}After the first milestone was done, the client wanted to collaborate more. So I setup another script which calculates data for candlestick-type chart (low, high, open, close, buy ratio) for intervals of N hours. Under the hood it was a wrapper around a fairly complex SQL query which can calculate the results incredibly fast even for millions of rows.{% endblocktrans %}

{% blocktrans %}All-in-all it was a really cool project, the client was great and I'd love to work on similar projects in the future.{% endblocktrans %}
